libf/phymars

文件定位

initracer.F90 定义模块 initracer_mod 和唯一例程 initracer(ngrid,nq)。它位于 tracer 配置解析之后、物理过程使用 tracer 之前：tracer_mod 已经持有 noms(:)、mmol(:)、radius(:)、rho_q(:)、alpha_lift(:)、alpha_devil(:) 等数组，本例程按 tracer 名称逐项识别 igcm_* 索引，填充分子量和物理属性，并根据 callkeys_mod / dust_param_mod 中的物理开关执行一致性检查。

本文件不是 traceur.def / tracer.def 解析器；配置文件读取由 tracer_mod、parse_traceur_file 和 parse_tracer_file 完成。initracer 的职责是把已经读入的名字映射为物理包约定的索引和属性，使尘埃、水、HDO、CO2 云、化学、热层和 NLTE 路径能通过 igcm_* 访问正确的 tracer。

共享状态与副作用

  • 本例程大量写入 tracer_mod 的模块状态，并通过 write(*,*) 打印识别结果、尘埃起尘参数和物性数组。
  • 它只按 noms(:) 字符串识别 tracer；名称大小写和拼写必须与源码中的字面量一致，例如 dust_mass、h2o_vap、ccnco2_mass、Ar_N2。
  • 若识别数量 count 不等于 nq，或某个开启的物理过程缺少必需 tracer，会调用 abort_physic 终止。
  • rho_q/radius/alpha_lift/alpha_devil 在物性初始化前先整体清零；后续分支只为相关 tracer 写非零值。

核心逻辑

  1. 数量和索引清零：若 nq<1 直接中止；随后把所有 igcm_* 索引重置为 0，避免旧状态残留。
  2. 尘埃族识别：按 dustbin>0 查找 dust01、dust02 等 bin 名称；按 doubleq 查找 dust_mass/dust_number；按 microphys 查找 ccn_mass/ccn_number；按 submicron、rdstorm、topflows 分别查找 dust_submicron、stormdust_* 和 topdust_*。
  3. 化学、水和离子名称识别：对所有 iq=1..nq 扫描 CO2、CO、O、H、HOx、NOx、CH4、N2、Ar、He、离子、电子、水汽/水冰、HDO、CO2 冰等名称，并写入对应 igcm_* 和 mmol。
  4. CO2 云附加 tracer 识别：若 co2clouds，识别 ccnco2_mass/number；若 meteo_flux，再识别 ccnco2_meteor_*；若 co2useh2o，识别 ccnco2_h2o_*。
  5. 全量匹配检查：要求 count == nq；否则打印已识别 tracer 并中止。这意味着配置中出现源码未知名称也会失败。
  6. HDO 顺序和父子关系检查：开启 hdo 时，要求 H2O 先于 HDO 初始化、HDO 作为 H2O 子 tracer，且 HDO tracer 位于列表末尾附近。
  7. 物性默认清零和密度常数：清零 rho_q/radius/alpha_lift/alpha_devil，设置 rho_dust=2500、rho_ice=920、rho_ice_co2=1650。
  8. 尘埃物性初始化：doubleq 分支检查 dust_mass/dust_number，设 nueff_lift=0.5，由 varian 推导 r3n_q/ref_r0，读取 reff_lift_dust，再设置 mass/number 的 radius 和起尘系数；rdstorm/topflows 复用 lifted dust 半径。非 doubleq 且 dustbin==1 时使用单粒径 3.e-6。
  9. 水、HDO、submicron、CCN 和 CO2 云物性：水汽/HDO 汽相半径为 0，水冰/HDO 冰半径 3.e-6、密度 rho_ice；CO2 冰半径 1.e-8、密度 rho_ice_co2；CCN 多数继承尘埃属性并把起尘系数设为极小值。
  10. 开关依赖检查：分别检查 water、hdo、microphys、co2clouds、rdstorm、topflows、callnlte/nltemodel、scavenging、photochem/callthermos 所需 tracer 是否存在。

伪代码

initracer(ngrid, nq):
    require nq >= 1
    reset all igcm_* indexes to 0

    for each enabled tracer family:
        scan noms(1:nq)
        if a known name is found:
            set igcm_* index
            set mmol when known
            append to identified list

    require identified_count == nq
    if hdo:
        require H2O before HDO, HDO as child tracer, HDO near end

    clear rho_q, radius, alpha_lift, alpha_devil
    set rho_dust, rho_ice, rho_ice_co2

    if doubleq:
        require dust_mass and dust_number
        derive varian, r3n_q, ref_r0
        read reff_lift_dust
        set dust mass/number radius and lifting coefficients
        optionally initialize stormdust/topdust
    else if dustbin == 1:
        set single-bin dust properties

    initialize scavenging CCN, submicron dust, water, HDO, CO2 cloud properties
    run feature-specific tracer dependency checks

参与的主题流程

主题 参与方式
尘埃循环 识别尘埃质量/数浓度、bin、submicron、stormdust、topdust tracer，并设置起尘系数、半径和密度。
水循环 识别 h2o_vap/h2o_ice、水云 CCN 和 HDO tracer，并检查水循环/微物理依赖。
CO2 循环 识别 CO2、CO2 冰、CO2 云 CCN、流星尘 CCN 和 H2O 来源 CO2 CCN。
高层大气/化学 识别 CO/O/H/N/离子/电子等化学与热层 tracer，并在 photochem、callthermos、callnlte 开启时做依赖检查。

写法特点

  • 该例程采用长串 if (noms(iq) == "...") 字符串匹配，而不是表驱动映射；新增 tracer 名称时必须同步维护识别、分子量、属性和依赖检查。
  • idtracers(count) 用于记录识别顺序；count /= nq 时会把已识别名称打印出来，便于定位配置中未知 tracer。
  • ngrid 在当前源码中未被使用，但保留在接口里。
  • meteo_flux 检查块中 igcm_ccnco2_meteor_mass 被连续检查两次，其中第一条错误消息提到 ccnco2_h2o_mass_ice；这是已在 CO2 tracer 段页标注过的文案/重复检查疑点。

复现要点

  • traceur.def / tracer.def 中每个 tracer 名称都必须被源码识别，否则 count /= nq 会中止。
  • 开启 doubleq 必须有 dust_mass 和 dust_number；开启 water 必须有 h2o_vap/h2o_ice；开启 microphys 必须有 ccn_mass/ccn_number。
  • 开启 co2clouds 至少需要 co2/co2_ice/ccnco2_mass/ccnco2_number；co2useh2o 和 meteo_flux 会进一步增加 tracer 要求。
  • reff_lift_dust 改变会影响 radius(dust_mass/dust_number)、alpha_lift 和 alpha_devil 的换算。
  • HDO tracer 的顺序和 parent/child 关系是硬检查，配置文件顺序不满足时会中止。

待确认

  • ngrid 在当前文件内未使用，是否保留给旧接口兼容或未来按网格初始化逻辑。
  • meteo_flux 检查块重复检查 igcm_ccnco2_meteor_mass 是否为有意冗余，或应有一项检查 igcm_ccnco2_meteor_number / 其它 H2O 来源 tracer。
