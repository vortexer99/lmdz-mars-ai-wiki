文件定位

ioipsl_getincom.F90 是无 IOIPSL 或本地 fallback 的 getin 实现，负责读取 .def 风格键值文件、保存 key 表、解析标量和数组。ioipsl_getin_p_mod.F90 位于 phy_common，把 getin 包装成并行安全的 getin_p：root 读取后广播到所有 MPI/OMP 参与者。

Mars 运行参与度：必经/条件经过。Mars 物理配置大量调用 getin_p；Mars 1D 和部分初始化工具直接调用 getin。datareadnc.F 也直接用 getin("datadir") 定位 surface.nc；该输入链见 Mars utility NetCDF 和 COMMON helper 边界。当编译使用 IOIPSL 时，getin_p 调 IOIPSL 的 getin；否则调本页的 ioipsl_getincom:getin。

getin fallback 流程

  1. 首次调用触发 getin_read，分配文本、key 表和 typed memory。
  2. getin_readdef 逐个打开配置文件，跳过空行和注释，读取 key = value。
  3. getin_decrypt 判断 key 类型，把 integer/real/character/logical 值放入对应存储区。
  4. getin_checkcohe 检查重复 key，源码注释和消息显示保留最后值。
  5. 各 getin* specific procedure 按调用方变量类型和维度回填；找不到 key 时保持调用方默认值。
  6. getin_dump 可输出已读数据库，用于 abort 时或调试时复现配置来源。

getin_p 并行契约

  • 每个 specific procedure 先执行 !$OMP BARRIER。
  • 只有 is_mpi_root .AND. is_omp_root 的线程调用 getin。
  • 标量直接 bcast(VarOut)；character 1D 逐元素广播；integer/real/logical 数组调用对应 bcast。
  • 因此 Mars 并行物理配置读取必须保证 mod_phys_lmdz_mpi_data、mod_phys_lmdz_omp_data 和 transfer/bcast 已初始化。

Mars 侧典型消费者

  • phymars/conf_phys.F 读取 ecritphy、outputs_per_sol、iphysiq、tauvis、callrad、water、co2clouds 等大量物理配置。
  • dynphy_lonlat/phymars/datareadnc.F 读取 datadir，决定 surface.nc 的输入目录；NetCDF 读取本身仍走 Mars 侧直接 NF_* API。
  • phymars/aeropacity_mod.F、watercloud_mod.F、initracer.F90 等使用 getin_p 获取局地风暴、微物理和 tracer 参数。
  • COMMON dyn3d/conf_gcm.F90、dyn3dpar/conf_gcm.F90 直接使用 ioipsl_getincom 或 IOIPSL getin 读取动力配置。

待确认
