calfis_p 与 call_physiq 接口契约

输入范围

dynphy_lonlat\calfis_p.F: L939-1079（call_physiq 调用和 nsplit_phys 循环）
dynphy_lonlat\phytitan.old\callphysiq_mod.F90: L1-97（Titan 参考实现）
dynphy_lonlat\phyvenus\callphysiq_mod.F90: L1-100（Venus 参考实现）
dyn3d_common\infotrac.F90: L5-26, L667-770（nqtot/tname/niadv 定义与初始化）

Mars 运行参与度：条件经过。Mars 并行 GCM 在 apphys=.TRUE. 时经过 calfis_p → call_physiq → physiq。Mars 的 callphysiq_mod 不在 COMMON 树中（位于 Mars 物理目录），但接口签名与 Titan/Venus 版本一致。

例程定位

本页聚焦 calfis_p 调用 call_physiq 时的接口契约：哪些参数被传入、哪些被返回、nqtot/tname/niadv 的语义、以及 nsplit_phys 时间分割机制。网格映射细节见 calfis-p-grid-mapping；整体调用链见 leapfrog-p-physics-coupling。

nqtot / tname / niadv 的关系

nqtot（tracer 总数）

nqtot 由 infotrac_init 计算，包含所有需要输送的 tracer：基本示踪物、高阶矩、水汽、液水等。它是动力和物理共享的 tracer 维度上限。

tname（tracer 名称数组）

tname(nqtot) 在 infotrac_init 中填充，每个 tracer 对应一个短字符串（如 "H2O_vap"、"dust_1" 等）。物理侧通过 tname 按名查找特定 tracer 以执行 specialized 过程（如辐射吸收、微物理）。

niadv（动力-物理 tracer 映射）

niadv(nqtot) 建立了动力 tracer 索引到物理 tracer 索引的对应关系：

! infotrac_init 中：
niadv(:) = 0
DO iq = 1, nqtot
   ! ... 对每个有 iadv(iq) > 0 的 tracer：
   niadv(iiq) = iq
ENDDO

niadv 的设计目的是允许动力和物理使用不同的 tracer 排序。然而在当前 calfis_p 中：

  • scatter 阶段按 iq=1,nqtot 顺序直接复制 pq(:,:,:,iq) → zqfi(:,:,:,iq)
  • gather 阶段同样按 iq=1,nqtot 直接复制 zdqfi(:,:,:,iq) → pdqfi(:,:,:,iq)
  • niadv 在 calfis_p 中未被使用

这意味着当前实现假设动力和物理使用相同的 tracer 排序。如果未来引入不同排序，需要在 scatter/gather 中加入 niadv 重映射。

nsplit_phys 时间分割

zdt_split = dtphys / nsplit_phys

DO isplit = 1, nsplit_phys
   jH_cur_split = jH_cur + (isplit-1) * dtvr / (daysec * nsplit_phys)
   debut_split = debut .AND. isplit == 1
   lafin_split = lafin .AND. isplit == nsplit_phys

   CALL call_physiq(klon, llm, nqtot, tname, ...)

   ! 用当前 tendency 更新状态（给下一个 split 用）
   zufi_omp = zufi_omp + zdufi_omp * zdt_split
   zvfi_omp = zvfi_omp + zdvfi_omp * zdt_split
   ztfi_omp = ztfi_omp + zdtfi_omp * zdt_split
   zqfi_omp = zqfi_omp + zdqfi_omp * zdt_split

   ! 累积 tendency
   zdufic_omp = zdufic_omp + zdufi_omp
   ! ... 同理其他
ENDDO

! 平均 tendency
zdufi_omp = zdufic_omp / nsplit_phys
! ... 同理其他

关键语义：

  • nsplit_phys 由 control_mod 提供，允许物理时间步内细分。
  • 每个 split 后，物理输入状态被当前 tendency 更新（前向 Euler），使下一个 split 看到更新后的场。
  • 最终输出的 tendency 是所有 split 的平均值。
  • debut_split 只在第一个 split 为真（触发物理初始化）；lafin_split 只在最后一个 split 为真（触发物理收尾）。

pducov 的特殊处理

pducov 是唯一的"非 OMP 切片"参数：

REAL, INTENT(IN) :: pducov(nbp_lon+1, nbp_lat, llm)

它的维度是 nbp_lon+1 × nbp_lat（物理侧全局网格维度），而不是 klon_omp。在 calfis_p 的 call_physiq 调用中，pducov 被原样传入而未按 OMP 线程切片。这意味着每个 OMP 线程的物理调用都接收完整的 pducov 数组。

这个设计的可能原因：

  1. 物理参数化可能需要全局视野的协变风 tendency 信息（如潮汐摩擦）。
  2. pducov 可能只在特定行星物理中被使用（Titan/Venus），Mars 物理可能忽略此参数。

callphysiq_mod 内部分派

Titan 和 Venus 的 callphysiq_mod 实现相同模式：

  1. firstcall 时验证 planet_type 匹配。
  2. 直接转发到 physiq_mod:physiq（行星特定的物理包）。
  3. Titan 版向 physiq 传递 zrfi_omp（涡度），Venus 版不传（注释掉了 plevmoy/tmoy）。

Mars 的 callphysiq_mod 不在 COMMON 树中，位于 Mars 物理目录。根据 COMMON 中 calfis_p.F 的注释代码块（L957-L1066），旧版曾按 planet_type 分支直接调用不同签名的 physiq，现已被统一的 call_physiq 接口替代。

契约约束总结

契约项 约束
klon 一致性 call_physiq 的 klon 必须等于调用方的 klon_omp
nqtot 一致性 动力和物理必须使用相同的 nqtot 值
tname 稳定性 tname 在运行期间不变；物理按名查找 tracer
tendency 单位 所有输出 tendency 为 */s（每秒变化率）
tendency 维度 输出 tendency 的 tracer 维与输入 zqfi 对齐（nqtot）
状态更新 物理可以假定输入状态是上一步物理输出更新后的值
pducov 范围 物理侧不应假定 pducov 与 klon_omp 有直接索引关系

复现要点

  • Mars 物理的 callphysiq_mod 接口必须与 COMMON 中 calfis_p 的调用签名严格匹配。修改接口时需要同步更新 Mars 物理目录中的 callphysiq_mod。
  • niadv 当前未使用意味着 tracer 在动力和物理间的排序必须一致。如果引入新的 tracer 类型，需确认 infotrac_init 产生的排序与物理侧预期一致。
  • nsplit_phys > 1 时，split 间的状态更新是前向 Euler（一阶），这在大 tendency 或小 split 数时可能引入误差。
  • pducov 的全局维度传递意味着它不能被 OMP 线程安全地按列访问——物理侧如果读取此数组需要注意线程安全。
